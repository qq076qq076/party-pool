# 網頁派對遊戲規格書（MVP v0.2）

## 1. 專案概述

### 1.1 專案名稱（暫定）
- `Party Pool：試膽派對`

### 1.2 專案目標
- 製作一款「主畫面 + 手機控制器」的多人網頁派對遊戲。
- 遊戲節奏與體驗參考《雞膽2─試膽挑戰─》（TORIDAMA2）類型：短回合、緊張感、派對互動。
- 玩家可透過掃描 QR Code 或輸入房間碼快速加入。

### 1.3 目標平台
- 主畫面（Host/Display）：桌機瀏覽器（Chrome/Edge）全螢幕執行 Godot Web 版。
- 控制器（Player Controller）：手機瀏覽器（iOS Safari / Android Chrome）。

---

## 2. 產品定位與範圍

### 2.1 核心定位
- 同地同樂（Couch Party）多人互動遊戲。
- 主畫面負責顯示遊戲內容；每位玩家以手機作為專屬控制器。
- 整體主題：`生存力測試`。

### 2.2 本階段範圍（MVP）
- 建立房間
- 加入房間（QR + 房間碼）
- 房間等待室（顯示人數、玩家清單、連線狀態）
- 反應型 mini game（第一關）
- 回合結束與結果畫面

### 2.3 暫不納入（Post-MVP）
- 帳號系統/好友系統
- 語音聊天
- 排行榜/賽季制
- 複雜配對（跨區自動配房）

### 2.4 美術方向（已定案）
- 整體風格：可愛 + 搞怪的日式派對風格
- 呈現目標：角色與介面辨識度高、表情誇張、色彩明亮

---

## 3. 使用者角色

- `房主（Host）`
  - 建立房間、顯示 QR 與房間碼、進入回合準備階段。
- `玩家（Player）`
  - 透過手機加入房間、在每回合作為控制器輸入操作。

---

## 4. 核心使用流程

### 4.1 主流程
1. 進入首頁，看到 `開房間` 與 `加入遊戲`。
2. 房主點 `開房間`：
   - 系統建立房間並生成房間碼（例：`ABCD`）。
   - 顯示房間內畫面：房間碼、QR Code、目前人數。
3. 玩家以手機：
   - 掃 QR Code 直接進入加入頁（帶入房間碼），或
   - 手動輸入房間碼加入。
4. 玩家輸入暱稱並同意必要權限後進房，主畫面即時更新人數與清單。
5. 房主按 `開始遊戲` 後進入「回合準備階段」（不會立刻開局）。
6. 顯示玩法說明與 `OK` 按鈕，系統啟動 60 秒準備倒數。
7. 若全員在 60 秒內按 `OK`，立即開局；若有人未按 `OK` 或斷線，60 秒到仍自動開始。
8. 房主不能手動強制開始。
9. 回合開始後玩家用手機操作，主畫面顯示結果。
10. 顯示最終結果，可選 `再來一局` 或 `返回大廳`。

### 4.2 例外流程
- 房間碼錯誤：提示「房間不存在」。
- 房間已滿：提示「房間已滿」。
- 遊戲已開始：提示「本局進行中，可排隊下一場（可選）」。
- 準備階段玩家斷線：倒數不中止，等 60 秒結束自動開局。
- 回合進行中玩家斷線：該回合不補時，下一回合可重連。

## 5. 功能需求

### 5.1 首頁
- 按鈕：`開房間`、`加入遊戲`
- 簡短說明：手機掃碼加入、建議開啟音效

### 5.2 開房間
- 建立唯一房間 ID 與短房間碼
- 顯示：
  - 房間碼（大字）
  - QR Code（可掃描）
  - 目前人數/最大人數（例：`3 / 8`）
  - 玩家名單（含連線狀態）
- 房主操作：
  - 開始遊戲（進入準備階段）
  - 關閉房間（可選）

### 5.3 加入遊戲（手機）
- 方式 A：掃碼（URL 含 room_code）
- 方式 B：手動輸入房間碼
- 輸入暱稱
- 必須接受必要權限後才能進入房間
  - MVP 必要權限：動態感測（DeviceMotion，用於搖晃關卡）
- 若拒絕權限：提示原因並停留在加入前頁

### 5.4 房間管理
- 房間碼格式：4~6 位英數大寫（MVP 建議 4 位）
- 每房最大人數：8（MVP 可調為 4~8）
- 房間生命周期：
  - 建立後 10 分鐘未開始自動釋放
  - 結束後 5 分鐘保留可重開

### 5.5 手機控制器（依關卡動態切換）
- 本規格先定義「輸入能力」，不綁定固定按鍵版型。
- 每個關卡於開始時下發該關卡的 `control_profile`，控制器 UI 依設定切換。
- MVP 輸入能力範圍：
  - 點按（Tap）/ 連打（Mash）/ 長按（Hold）
  - 滑動（Swipe）
  - 搖晃（Shake，使用裝置動態感測）
- 狀態顯示：
  - 連線中 / 已連線 / 斷線重連
  - 當前回合操作提示（依關卡顯示）
  - 感測器狀態（可用 / 未授權 / 不支援）

### 5.6 遊戲回合（MVP：第一關）
- 主題：生存力測試
- 第一個 mini game：固定時間連打（Tap Challenge）
- 規則：在固定時間內，點擊次數最多者為該關第一名
- 回合長度：20 秒（可配置）
- 回合流程：
  - 玩法說明畫面（顯示規則、操作方式、目標）
  - 玩家準備確認（每位玩家手機按 `OK`）
  - 準備階段最長 60 秒（全員 `OK` 立即開始；逾時自動開始）
  - 下發本回合控制設定（`control_profile`）
  - 倒數開始（3 秒）
  - 進行中（主畫面呈現、手機輸入）
  - 結算（排名與加分）
- 計分規則：
  - 每關只給第一名 +1 分
  - 若第一名同分，並列第一者都 +1 分
- 局數：3 回合（可配置）

### 5.7 語言支援
- 支援語言：English / 繁體中文
- 預設語言：依瀏覽器語言自動判斷
  - `zh-TW` 或 `zh-*` -> 繁體中文
  - 其他 -> English
- 玩家可在首頁與手機控制器頁手動切換語言
- 系統記住同裝置上次語言選擇（local storage）

### 5.8 重連與身份綁定（提案）
- 玩家首次入房後由伺服器發放 `rejoin_token`（僅本房間有效）
- 手機端將 `rejoin_token` 暫存 local storage
- 斷線後重開頁面可用 `room_code + rejoin_token` 自動找回原玩家席位
- 準備階段斷線不影響開局時間：仍以 60 秒規則自動開局
- 回合進行中斷線：該玩家該回合停止累計，下一回合可恢復

## 6. 非功能需求

### 6.1 效能
- 控制輸入到主畫面反應延遲：目標 < 120ms（同區網路）
- 主畫面 FPS：目標 60 FPS（最低可接受 30 FPS）
- 支援同房 8 人穩定連線

### 6.2 相容性
- 主畫面：Chrome/Edge 最新兩個大版本
- 手機：iOS Safari、Android Chrome 最新兩個大版本
- 直橫向：
  - 主畫面固定橫向
  - 手機控制器建議直向（可鎖）

### 6.3 可用性
- 掃碼到加入成功步驟 <= 3 步
- 房間碼需高對比、可遠距辨識
- 斷線需可自動重連（同裝置同瀏覽器）

---

## 7. 技術架構（建議）

### 7.1 前後端組成
- `Godot Web 客戶端（主畫面）`
  - 遊戲畫面、房間顯示、回合控制
- `Mobile Web 控制器頁（HTML5）`
  - 玩家輸入 UI、連線狀態、回合提示
- `即時連線伺服器（WebSocket）`
  - 房間管理、玩家連線、事件轉發

### 7.2 連線協定（事件制）
- Client -> Server
  - `create_room`
  - `join_room`
  - `permission_granted`
  - `player_ready_ok`
  - `player_input`
  - `motion_permission_result`
  - `request_rejoin`
  - `start_game`
  - `heartbeat`
- Server -> Client
  - `room_created`
  - `room_state_updated`
  - `ready_timer_started`
  - `ready_status_updated`
  - `ready_timeout_reached`
  - `game_started`
  - `round_started`
  - `control_profile_updated`
  - `round_result`
  - `error`

### 7.3 狀態同步策略
- 伺服器權威（Server Authoritative）：
  - 房間狀態、回合開始/結束、分數由伺服器決定
- 客戶端僅負責輸入與渲染，避免作弊與狀態分歧

### 7.4 通訊與部署安全
- 正式環境必須使用 `HTTPS`
- 即時連線必須使用 `WSS`（WebSocket Secure）
- 不允許混合內容（HTTPS 頁面呼叫 WS）

## 8. 資料模型（MVP）

### 8.1 Room
- `room_id`: string（UUID）
- `room_code`: string（短碼）
- `host_socket_id`: string
- `status`: `waiting | readying | playing | ended`
- `max_players`: number
- `players`: Player[]
- `ready_deadline_at`: timestamp
- `created_at`: timestamp

### 8.2 Player
- `player_id`: string
- `nickname`: string
- `socket_id`: string
- `is_connected`: boolean
- `ready_status`: `pending | ok`
- `rejoin_token_hash`: string
- `score`: number
- `last_seen_at`: timestamp

---

## 9. UI 畫面清單

1. 首頁（開房間 / 加入遊戲）
2. 開房後房間頁（房間碼 + QR + 玩家名單 + 人數）
3. 手機加入頁（房間碼輸入/帶入 + 暱稱 + 權限同意）
4. 手機等待頁（連線成功、等待開始）
5. 回合說明 + OK 準備頁（含 60 秒倒數）
6. 遊戲回合頁（主畫面）
7. 結算頁（單回合/最終）

---

## 10. 安全與風險

- 房間碼防爆破：
  - 同 IP 短時間輸入失敗次數限制
- 基本防濫用：
  - 暱稱長度限制與敏感字過濾（可後補）
- 連線風險：
  - 行動網路切換造成斷線，需要重連邏輯
- 權限風險：
  - 使用者拒絕動態感測權限會導致無法入房

---

## 11. 里程碑規劃

### M1：連線與房間（1~2 週）
- 首頁、開房、加房、QR 顯示、房間同步
- 權限同意流程與拒絕處理

### M2：控制器與回合原型（1~2 週）
- 手機輸入事件、Godot 角色/物件反應、回合開始/結束
- 60 秒 OK 準備倒數流程

### M3：結算與可玩版本（1 週）
- 計分、結果頁、重開局流程、基本音效與 UI 拋光
- 瀏覽器語言預設與手動切換

---

## 12. 驗收標準（MVP Done）

- 可由主畫面成功建立房間並看到房間碼與 QR Code
- 至少 2~8 位玩家可透過手機加入同一房間
- 加入房間前必須完成必要權限同意
- OK 準備最長 60 秒，逾時可自動開始回合
- 房主無法手動強制開始回合
- 玩家手機輸入可即時控制主畫面互動
- 可完整跑完一局（至少 3 回合）並顯示結果
- 每關第一名 +1，同分第一皆 +1
- 斷線後可使用 token 重連回同房間
